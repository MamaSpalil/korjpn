# Промт: Извлечение офсетов из main.exe v1.04.10

---

## Системный промт (System Prompt)

Ты — эксперт по реверс-инжинирингу игровых клиентов, специализирующийся на анализе бинарных файлов с помощью IDA Pro, x64dbg, Cheat Engine и аналогичных инструментов. Твоя задача — систематически извлекать и каталогизировать все значимые адреса памяти (офсеты) из исполняемого файла `main.exe` версии **1.04.10**.

---

## Пользовательский промт (User Prompt)

Проведи полный анализ файла `main.exe` версии 1.04.10 и создай исчерпывающий каталог всех офсетов памяти, сгруппированных по категориям. Для каждого офсета укажи:

- **Адрес** (hex, базовый + относительный при необходимости)
- **Тип данных** (DWORD, FLOAT, BYTE, STRING, POINTER и т.д.)
- **Размер** в байтах
- **Описание** назначения
- **Метод обнаружения** (паттерн байт, символ, строка и т.д.)
- **Проверочное значение** или диапазон допустимых значений, если применимо

## Категории офсетов для поиска

### 1. СОСТОЯНИЕ СОЕДИНЕНИЯ И АВТОРИЗАЦИЯ
```
- Режим выбора сервера:
  * Текущий выбранный сервер (индекс)
  * Список серверов (указатель на массив)
  * Статус подключения к серверу
  * IP-адрес сервера (строка)
  * Порт сервера

- Экран ввода логина/пароля:
  * Буфер логина (char[])
  * Буфер пароля (char[])
  * Флаг успешной авторизации
  * Код ошибки авторизации
  * Статус сессии (активная / неактивная)
  * Флаг активного аккаунта (account active flag)
```

### 2. ВЫБОР ПЕРСОНАЖА
```
- Индекс выбранного слота персонажа
- Имя выбранного персонажа (char[])
- Класс персонажа (byte — warrior, wizard, elf и т.д.)
- Уровень персонажа на экране выбора
- Количество ресетов (resets count)
- Координаты персонажа на карте предпросмотра
- Флаг блокировки слота персонажа
```

### 3. СТАТИСТИКА ПЕРСОНАЖА В ИГРЕ
```
- HP текущее (current health points)
- HP максимальное (max health points)
- Мана текущая (current mana)
- Мана максимальная (max mana)
- Стамина/выносливость текущая (current stamina/AG)
- Стамина максимальная (max stamina/AG)
- Опыт за убийство (experience per kill — последнее значение)
- Опыт текущий (current total experience)
- Опыт до следующего уровня (experience to next level)
- Уровень персонажа (character level)
- Очки силы (strength)
- Очки ловкости (agility)
- Очки живучести (vitality)
- Очки энергии (energy)
- Очки команды (command — для DL)
- Свободные очки статов (free stat points)
- Количество ресетов (resets)
- Мастер-уровень (master level)
- Очки мастер-левела (master exp/points)
```

### 4. ПОЗИЦИЯ И НАВИГАЦИЯ
```
- X-координата персонажа (float/int)
- Y-координата персонажа (float/int)
- Z-координата (высота, если применимо)
- ID текущей карты/локации (map ID)
- Название текущей карты (string pointer)
- Флаг телепортации (teleport in progress)
- Целевые координаты телепортации (target X, target Y)
- Таблица локаций/варпов (pointer to warp table)
- Направление движения (direction byte)
- Скорость передвижения
```

### 5. БОЕВАЯ СИСТЕМА И МОНСТРЫ
```
- Указатель на массив/список монстров на карте
- Структура объекта монстра:
  * ID монстра
  * Тип монстра (monster type ID)
  * Имя монстра (string)
  * HP монстра текущее
  * HP монстра максимальное
  * X-координата монстра
  * Y-координата монстра
  * Статус монстра (живой/мёртвый/агрессивный)
  * Уровень монстра

- Атака монстра:
  * Флаг атаки монстра (monster attacking flag)
  * Урон монстра по игроку (last damage taken)
  * Цель атаки монстра (target ID)

- Убийство монстров:
  * Счётчик убитых монстров (kill counter)
  * ID последнего убитого монстра
  * Флаг смерти монстра
  * Таймер респауна
```

### 6. ПРЕДМЕТЫ И ИНВЕНТАРЬ
```
- Указатель на инвентарь персонажа
- Структура предмета (item struct):
  * ID предмета (item ID)
  * Тип предмета (item type/category)
  * Уровень предмета (+level)
  * Опции предмета (skill, luck, option)
  * Серийный номер предмета
  * Количество (для stackable)
  * Позиция в инвентаре (slot X, slot Y)
  * Флаг экипировки

- Подбор вещей (item pickup):
  * Флаг подбора предмета
  * ID подбираемого предмета
  * Позиция дропа на карте

- Дроп на земле:
  * Массив предметов на земле (ground items array)

```

### 7. СУНДУК / ХРАНИЛИЩЕ (VAULT/WAREHOUSE)
```
- Флаг открытого сундука (vault open flag)
- Указатель на массив предметов в сундуке

- Флаг блокировки сундука
- Передача предмета в сундук:
  * Флаг передачи (transfer to vault flag)
  * ID передаваемого предмета
  * Слот назначения в сундуке
- Забор предмета из сундука:

  * Слот источника в сундуке
```

- Флаг PvP режима (pvp flag)

  * ID игрока

- Указатель на массив квестов

## Формат вывода результатов

Представь результаты в виде структурированной таблицы следующего вида:

```
КАТЕГОРИЯ: [Название категории]
┌─────────────────────┬────────────┬──────────┬────────────────┬──────────────────────────────────┐
│ Название офсета     │ Адрес (HEX)│ Тип      │ Размер (байт)  │ Описание / Примечание            │
├─────────────────────┼────────────┼──────────┼────────────────┼──────────────────────────────────┤
│ CurrentHP           │ 0x00XXXXXX │ DWORD    │ 4              │ Текущий HP персонажа (0–MaxHP)   │
│ MaxHP               │ 0x00XXXXXX │ DWORD    │ 4              │ Максимальный HP                  │
│ ...                 │ ...        │ ...      │ ...            │ ...                              │
└─────────────────────┴────────────┴──────────┴────────────────┴──────────────────────────────────┘
```

После таблицы укажи:
- **Byte pattern** для нахождения каждого офсета (AOB scan signature)
- **Pointer chain**, если офсет доступен через цепочку указателей (base + offset1 + offset2 + ...)
- **Примечания по версии**: что изменилось по сравнению с предыдущими версиями (если известно)

## Инструкции по методологии поиска

При анализе используй следующий порядок:

1. **Статический анализ** (IDA Pro / Ghidra):
   - Ищи строковые ссылки на имена параметров ("HP", "Mana", "Level", "Reset" и т.д.)
   - Анализируй функции инициализации структур персонажа
   - Ищи паттерны обработки сетевых пакетов (recv/send handlers)

2. **Динамический анализ** (Cheat Engine / x64dbg):
   - Сканируй значения во время игры (exact value scan)
   - Используй изменение значений для подтверждения (пей зелье → сканируй изменение HP)
   - Трассируй указатели от базового адреса

3. **Паттерн-поиск** (AOB Scan):
   - Для каждого найденного офсета указывай байтовую сигнатуру ±16 байт вокруг адреса
   - Маскируй изменяемые байты символом `??`
   - Пример: `8B 0D ?? ?? ?? ?? 85 C9 74 ?? 8B 01` → указатель на структуру персонажа

4. **Верификация**:
   - Проверяй каждый офсет на нескольких сессиях
   - Убеждайся в стабильности адресов при перезапуске клиента
   - Помечай статические и динамические (pointer-based) адреса

## Дополнительные требования

- Если офсет является **динамическим** (меняется при перезапуске), предоставь полную цепочку указателей начиная от статического базового адреса.
- Для **структур объектов** (монстры, игроки, предметы) укажи полный layout структуры с размером каждого поля.
- Укажи **размер массива** для списков (максимальное количество монстров на экране, максимальный размер инвентаря и т.д.).
- Отдельно отметь офсеты, которые **не удалось найти** или требуют дополнительного исследования.
- Укажи **версию анализируемого файла** (MD5/SHA256 хэш, дата сборки из PE-заголовка).
